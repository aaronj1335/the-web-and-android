Differences

  • Let's focus on differences that point to discrepancies in high-level priorities.
  • Based on that we'll get to the point of this talk: what each can learn from the other.

#1 Origins

  • The web started as a way to share research
  • Android started as a way to make better mobile software
  • Sir Tim Berners Lee was working at CERN, a European nuclear research facility, and he invented the web for sharing research information.
  • Andy Rubin talked about Android as a way to make smarter mobile devices.
  • While people weren't using the word "apps" initially, it was definitely about expanding the possibilities of software applications, given a powerful device you carried everywhere with you.
  • I think we still see echoes of these origins today.
  • Can anybody give examples?
  • Web search is still better than mobile app indexing
  • On the other hand, how many of us use a mobile web app for chat?

#2 What do we call content makers?

  • Each platform talks about content makers (newspapers, social media influencers, Netflix, etc) by different (telling) names.
  • Web standards refer to them as "authors"
  • Android docs refer to them as "publishers"
  • What's the first thing that pops into your head when you hear these words?
  • Authors: some person hunched over a typewriter, working on their next novel.
  • Publishers: companies in Manhattan that pay the authors.
  • Basically what Buddy the Elf's dad does.
  • This speaks to each platforms target audience.
  • The web targets on a finer granularity, so you get a lower barrier to entry.
  • Android is going after organizations with more resources, so it's not as easy to pick up, but you can make higher-fidelity user experiences.

#3 What's the authority?

  • The web: DNS + CA
  • Android: OEM + app store
  • The web has very distributed and loose authority.
  • Android is defined by a much smaller set of players.
  • A general theme for platforms: the more centralized authority, the faster they evolve.

What can the web learn from Android?

#1 List views are the only layout that matters.

  • It's the most important layout for mobile. For example, 80% of the apps on my home screen start with a list view. How many on yours?
  • Thing is, list views are hard. Android had to rewrite theirs, but not before Facebook went to a ton of trouble authoring their own..
  • Any viable mobile platform should make this easy on developers.
  • The web actually has pretty advanced layout tools (CSS flex, grid), but there's still no standard way to do an efficient list view.

#2 You still need shared memory multi-threading.

  • Low-end devices are stuck in 2014.
  • But we keep writing software that expands to fill the resources of the phones at the top of the ever-widening performance gap (see page weight over time).
  • As long as these are Von Neumann machines, the only viable option for things like declarative UI (React, etc), immutable data structures, realtime data, etc. is multi-threading that allows proper memory sharing (not just message passing).

#3 Battery matters.

  • In the past, the web didn't drain battery because it didn't really do stuff in the background or work offline.
  • As the web tries to make proper mobile apps (messaging apps, social media feeds, etc), it needs to get better at background and offline.
  • Android not only has WorkManager, its 3rd iteration of a battery-saving background scheduler, it's surfacing battery consumption to the user via settings and system tray notifications.
  • The web isn't really having these discussions.

What can Android learn from the web?

#1 Lower the barrier to entry.

  • The ease of making a website is sometimes viewed pejoratively, but this makes the ecosystem more diverse!

#2 Deliver apps incrementally.

  • Average web page is 10% average Android app.
  • Web has super flexible options for loading code incrementally, sharing code across domains, and caching subsets.
  • Android app bundles seem to be the first "official" means for this. Advanced apps (i.e. Google Play services, Facebook, etc) have their own systems for this.
  • Embedding webviews for shared logic is common (even in big apps!).
  • Tooling is bad everywhere though. Hell is another platform's build system.

#3 Make it async, not concurrent.

  • Concurrency is doing things at the same time.
  • Async is doing something (file I/O, network, etc) without blocking on the result.
  • Most of what apps do is I/O bound.
  • If you want a responsive app, you need to do I/O without blocking the main thread.
  • Android historically has reached for concurrency, the web for async.
  • Concurrency is more complicated though, and Android hasn't made it easy to do async-but-not-concurrent.
  • This is changing with Kotlin coroutines.

#4 Experiment with DX more.

  • The web re-invents the same wheels a lot.
  • This can be frustrating
  • But the web is really good at incubating developer experiences as a result.
  • It's so easy to deploy apps that you get shootouts like TodoMVC and Hacker News PWA, and jsPerf.
  • A lot of DX ideas come from the web.
  • This is a product of so many differnt groups trying to do the same thing.
  • It can be overwhelming, but there's a reason Android is implementing the web's UI approach and not visa-versa.

The web and Android are the best platforms ever (objectively)

There's still a lot they could learn from eachother.

Let's consider this and participate actively in these platforms.
